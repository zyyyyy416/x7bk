## 1. 产品概述

### 1.1 产品定位

**小7记账**是一款以"极简操作、深度分析"为核心理念的共享记账 App。聚焦个人日常记账与小型共享账本（家庭、情侣、室友）两大核心场景，通过智能分类、自动统计与趣味指标（恩格尔系数），让用户"记最少的账，看最透彻的自己"。

### 1.2 产品愿景

> 让每一笔花销都有意义 —— 不只是记录数字，更是理解生活。

### 1.3 核心差异化

| 维度 | 竞品现状 | 小7记账策略 |
|------|----------|------------|
| 操作体验 | 表单字段多、步骤繁琐 | **3秒极速记账**：金额+分类，两步完成 |
| 分析深度 | 基础饼图/柱状图 | 多维度交叉分析 + 趋势预测 |
| 趣味性 | 冷冰冰的数字报表 | **恩格尔系数**人格化解读，让数据有温度 |
| 协同记账 | 大多仅支持单人 | 原生共享账本，实时同步，权限分明 |

### 1.4 产品名称含义

- **小7** = "小"代表轻量极简，"7"代表一周七天，寓意"日日记账，天天清晰"
- 英文名: **X7BK** (X7 Bookkeeping)

### 2.1 用户画像

## 3. 核心功能模块

### 3.1 极简记账引擎

#### 3.1.1 记账流程（核心路径）

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ ① 输入金额 │ → │ ② 选择分类 │ → │ ③ (可选)备注 │ → │ ④ 完成记账 │
│  数字键盘  │    │ 智能推荐  │    │  拍照/语音  │    │  动画反馈  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
    0.5s            1.5s            1.0s            done!
```

#### 3.1.2 功能要点

- **智能金额输入**: 大数字键盘，支持小数点后两位；支持简单四则运算（如 `28+15` 自动计算）
- **分类智能推荐**: 基于时间（早餐时段推荐"早餐"）、地点、历史习惯，智能排序常用分类
- **快速备注**: 预设常用备注模板（如"午饭""打车""买菜"），支持语音转文字备注
- **拍照记账**: 支持拍摄小票/发票，OCR 自动识别金额与商户名
- **多账本切换**: 首页顶部一键切换"个人账本"与各个"共享账本"
- **收支类型**: 支出（默认）/ 收入（如工资、报销、红包）

#### 3.1.3 快捷操作

- **桌面小组件 (Widget)**: 一键进入记账页，预设常用分类快捷方式
- **通知栏快捷记账**: 下拉通知栏直接输入金额
- **最近记录一键复记**: 长按历史记录 → "再来一笔"

### 3.2 共享账本系统

#### 3.2.1 账本模型

```
账本 (Book)
├── 账本名称、封面、创建者
├── 成员列表 (Member[])
│   ├── 角色：管理员 / 普通成员
│   └── 权限：可查看 / 可记账 / 可管理
├── 账单列表 (Bill[])
│   ├── 金额、分类、备注、照片
│   ├── 记录人、时间戳
│   └── 归属类型：个人 / 共享
└── 结算周期 (可选)：月度 / 自定义
```

#### 3.2.2 功能要点

- **创建共享账本**: 通过邀请码/二维码邀请成员加入
- **实时同步**: 任一成员记账后，其他成员即时可见
- **消费归属**: 每笔消费可标记为"个人消费"或"共享消费"
- **简单结算**: 自动计算成员间应付/应收差额，支持"一键已平账"标记
- **账单备注与评论**: 支持对某笔账单留言互动（如"这顿我请！"）
- **账本归档**: 支持将已完成/不再使用的账本归档保存

### 3.3 预算管理

- 支持按**总支出**或**特定分类**设置月度预算
- 预算消耗进度条实时展示
- 超额时推送提醒（可配置阈值：80%/90%/100%）
- 支持**共享账本预算**：如"本月家庭超市采购预算 ¥2000"

## 4. 智能分类体系设计

### 4.1 设计原则

1. **MECE 原则**: 相互独立、完全穷尽，每笔消费有且仅有一个归属
2. **符合直觉**: 用户 3 秒内能找到目标分类
3. **覆盖现代消费**: 纳入外卖、奶茶、网购、知识付费、宠物等新时代消费场景
4. **可扩展**: 支持用户自定义添加二级分类

## 6. 恩格尔系数特色功能

### 6.1 计算模型

```
恩格尔系数 = (食品支出总额 ÷ 个人消费支出总额) × 100%

其中:
- 食品支出 = sum(一级分类="餐饮饮食" 下所有二级分类的支出金额)
- 个人消费支出总额 = sum(所有支出)
- 排除项（不计入分母）：转账、还款、投资等非消费性支出
```

### 6.2 判定标准（中国国情适配版）

| 恩格尔系数 | 等级 | 标签 | 趣味解读 |
|:----------:|:----:|------|----------|
| < 20% | ⭐⭐⭐⭐⭐ | **财务自由级** | "你已经不需要为食物操心啦！" |
| 20% - 30% | ⭐⭐⭐⭐ | **富足小康级** | "吃得好，活得也好，人生赢家！" |
| 30% - 40% | ⭐⭐⭐ | **小资舒适级** | "恩格尔系数刚刚好，生活有滋有味。" |
| 40% - 50% | ⭐⭐ | **温饱务实级** | "干饭人，干饭魂！吃是人生大事。" |
| > 50% | ⭐ | **终极吃货级** | "你的胃就是你的财富管理中心！🏆" |

> **参考**: 根据国家统计局数据，中国城镇居民恩格尔系数约为 28%-30%。因外卖/餐饮消费习惯差异，App 内的指标更多作为**趣味参考**，不作为严谨的经济学评估。

### 6.3 功能呈现

- **实时仪表盘**: 首页卡片展示当前月恩格尔系数 + 等级标签
- **趋势追踪**: 折线图展示近 12 个月恩格尔系数变化趋势
- **趣味解读**: 每月生成一条个性化评语
- **社交分享**: 支持生成"恩格尔系数名片"分享到微信/朋友圈
- **等级成就**: 连续3个月保持"富足小康级"解锁专属成就徽章

## 7. UI/UX 设计原则

### 7.1 设计语言

- **风格**: 极简清新，扁平化设计
- **主色调**: 薄荷绿 `#00B894` + 暖白背景，传递"轻量、活力、健康"的财务观
- **字体**: 数字使用等宽字体（Tabular Lining），确保金额对齐
- **图标**: 统一使用线性图标，二级分类使用彩色 emoji

### 7.2 交互原则

1. **3秒法则**: 核心记账流程不超过3步、不超过3秒
2. **拇指热区**: 主要操作区域位于屏幕下半部，方便单手操作
3. **防误操作**: 删除账单需二次确认；大额支出（>¥1000）需确认
4. **无网络可用**: 核心记账功能离线可用，联网后自动同步
5. **无障碍**: 支持大字体模式、高对比度模式

### 7.3 信息架构

```
Tab Bar
├── 🏠 首页     → 当月概览卡片 + 恩格尔系数 + 最近账单列表
├── ➕ 记账     → (居中突出按钮) 金额输入 → 分类选择 → 完成
├── 📊 分析     → 图表分析 + 报表中心
├── 📒 账本     → 共享账本列表 + 成员管理
└── 👤 我的     → 个人设置 + 分类管理 + 预算 + 数据导出
```

## 8. 技术栈选型方案

### 8.0 决策前提

> **已确认**:
> - ✅ 目标平台 Android，**未来预留 iOS 扩展能力**
> - ✅ **共享账本**是核心功能，需要云端同步
> - ✅ **成本控制**: 优先使用免费云服务层，**¥0/月** 上线

基于以上约束，跨平台框架 (React Native) + 免费 BaaS 是唯一最优解。以下是一主一备两套方案。

---

### 8.1 方案对比总览

| 维度 | 方案A: RN + Supabase 🔥 | 方案B: Kotlin Native + LeanCloud |
|------|:---:|:---:|
| **开发效率** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **iOS 扩展** | ✅ 一套代码直接复用 | ❌ 需完全重写 |
| **国内可用性** | ⭐⭐⭐ (自部署) | ⭐⭐⭐⭐⭐ |
| **云端成本** | **¥0** (免费层) | **¥0** (开发版免费) |
| **数据库能力** | PostgreSQL (强) | 文档型 (中) |
| **实时同步** | ✅ WebSocket 原生支持 | ✅ LiveQuery |
| **社区生态** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **包体积** | ~15-25MB | ~8-15MB |

---

### 8.2 🥇 方案A：React Native + Supabase（✅ 选定方案）

#### 技术组成

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | **React Native 0.76+** + Expo | 跨平台，一套代码覆盖 Android + iOS |
| 语言 | TypeScript | 类型安全，降低 bug 率 |
| 状态管理 | Zustand + React Query (TanStack) | 轻量状态 + 服务端缓存与离线支持 |
| UI 组件库 | React Native Paper | Material Design 3，开箱即用 |
| 图表库 | react-native-gifted-charts / Victory | 折线图、饼图、旭日图 |
| 本地存储 | WatermelonDB (基于 SQLite) | 离线优先，与后端自动同步 |
| 后端/BaaS | **Supabase** | 开源 BaaS，PostgreSQL + 实时同步 |
| 数据库 | PostgreSQL 15 (Supabase 内置) | 支持窗口函数、CTE，分析查询能力强 |
| 实时同步 | Supabase Realtime (WebSocket) | 共享账本实时推送 |
| 认证 | Supabase Auth | 手机号 OTP 登录 (国内可用) |
| 文件存储 | Supabase Storage (S3 兼容) | 账单图片 / 小票 OCR 图片 |

#### 💰 费用分析：Supabase 免费层

| 资源 | 免费额度 | 对小7记账是否够用？ |
|------|----------|:---:|
| 数据库空间 | **500 MB** | ✅ 一条账单 ~200B，够存 **250万条** |
| 每月 API 请求 | **无限** (有速率限制) | ✅ 个人+小团队使用绰绰有余 |
| 每月数据库流量 | **2 GB** | ✅ 日常同步完全够用 |
| 实时并发连接 | **200 个** | ✅ 用户量 < 500 时无压力 |
| 文件存储 | **1 GB** | ✅ 小票照片压缩后 ~100KB，可存 1万张 |
| 认证 (手机/邮箱) | **5万 MAU** | ✅ 正式商用前都够 |
| 项目数量 | **2 个** | ✅ dev + prod 正好 |

> 📌 **结论**: Supabase 免费层完全覆盖开发阶段 + 上线初期（数百用户级别）。只有当日活用户突破 **~500 人** 或存储超 **500MB** 时才需升级到 Pro ($25/月)。届时产品已有收入或可轻松承担。

#### 关于国内访问延迟

Supabase 官方托管在新加坡/美国，国内直连延迟 200-500ms。有两种解法：

| 解法 | 费用 | 说明 |
|------|:---:|------|
| **A. 先用官方免费层** (推荐 MVP) | ¥0 | 200ms 延迟对记账 App 影响不大（非实时游戏），异步同步体验可用 |
| **B. 自部署到阿里云/腾讯云** | ¥50-70/月 | 用 Supabase 开源版 Docker 部署到国内 ECS，延迟 < 20ms |

> 📌 **建议**: MVP 阶段先用 A（免费），验证产品后视延迟情况决定是否迁移到 B。

#### 为什么选 React Native 而非 Flutter？

| 因素 | RN | Flutter |
|------|:--:|:--:|
| JS/TS 生态 + 招聘 | ✅ 巨大优势 | ❌ Dart 小众 |
| 热更新 (CodePush) | ✅ 成熟方案 | ⚠️ 受限 |
| Supabase SDK 官方支持 | ✅ JS SDK 第一梯队 | ⚠️ Dart SDK 社区维护 |
| 国内第三方库兼容（微信SDK等） | ✅ RN 生态完善 | ❌ Flutter 插件常滞后 |
| 包体积 | 15-25MB | 20-30MB |

### 8.3 🥈 备选方案B：Kotlin Native + LeanCloud

> 仅在"确定不做 iOS 且追求极致 Android 体验"时考虑。

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | Kotlin + Jetpack Compose | Android 原生声明式 UI |
| 架构 | MVVM + Clean Architecture | 分层解耦 |
| 本地存储 | Room Database | SQLite 抽象层 |
| 后端/BaaS | LeanCloud | 国产 BaaS，国内低延迟 |
| 实时同步 | LeanCloud LiveQuery | WebSocket 推送 |

**优劣势**: 原生体验最佳、包体积最小、国内服务低延迟。但 **iOS 需重新开发**，成本翻倍。

## 10. 非功能性需求

### 10.1 性能要求

| 指标 | 目标值 |
|------|--------|
| 冷启动时间 | < 2 秒 |
| 热启动时间 | < 0.5 秒 |
| 记账页面打开 | < 300ms |
| 图表渲染 (含100条数据) | < 500ms |
| 列表滑动帧率 | 60fps |
| 离线数据同步延迟 | < 3 秒 (网络恢复后) |

### 10.2 安全要求

- 用户密码使用 bcrypt 哈希存储
- API 通信全链路 HTTPS / TLS 加密
- 数据库行级安全 (RLS)：用户只能访问自己的数据或被授权的共享账本数据
- 敏感操作（删除账本、移除成员）需二次确认
- 数据每日自动备份

### 10.3 兼容性

- Android 最低版本: Android 8.0 (API 26)
- 屏幕适配: 支持 4.7" - 6.9" 主流屏幕尺寸
- 支持暗黑模式 (Dark Mode)
- 支持中文简繁体

```
  id, book_id, user_id, role[admin|member], joined_at

  id, name, icon, parent_id, type[expense|income], engel_eligible, sort_order

Bill (账单)
  id, book_id, user_id, category_id, amount, type[expense|income],
  note, photo_url, bill_date, is_shared, created_at

Budget (预算)
  id, user_id, book_id(nullable), category_id(nullable),
  amount, period[monthly], start_date

Settlement (结算)
  id, book_id, from_user_id, to_user_id, amount, status, settled_at
```

### 11.2 关键 API 设计

```
POST   /auth/register          # 注册
POST   /auth/login             # 登录
GET    /books                  # 获取我的账本列表
POST   /books                  # 创建账本
POST   /books/:id/members      # 邀请成员
GET    /books/:id/bills        # 获取账单列表 (支持分页/筛选)
POST   /books/:id/bills        # 新增账单
PUT    /bills/:id              # 修改账单
DELETE /bills/:id              # 删除账单
GET    /analysis/summary       # 分析概览 (时间/账本维度)
GET    /analysis/engel         # 恩格尔系数历史
GET    /analysis/trend         # 分类趋势
POST   /export/csv             # 导出 CSV
```
